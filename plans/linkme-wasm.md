# Fork linkme for WASM Support

## Problem

`linkme` uses `#[distributed_slice]` which relies on platform-specific linker
sections (`.init_array` on Linux, `__DATA,__mod_init_func` on macOS). These don't
exist on `wasm32-unknown-unknown`, so miniR can't compile to WASM.

linkme is used for auto-registration of all 800+ builtins. Without it, we'd need
a manual registry — thousands of lines of boilerplate.

## Approach: Fork and Add WASM Backend

Clone https://github.com/dtolnay/linkme, add a WASM-compatible backend, vendor
the fork as a full dependency (not a crate registry dep).

### How linkme Works

1. `#[distributed_slice]` on a static defines a slice that collects items
2. `#[distributed_slice(REGISTRY)]` on items adds them to the slice
3. The proc macro emits linker-section attributes (platform-specific)
4. At program start, the linker/loader populates the slice from the sections

### WASM Backend Options

**Option A: `ctor` crate pattern — inventory at startup**

On WASM, there are no linker sections. Instead:
- The proc macro emits a `#[link_section = "..."]` on supported platforms
- On `wasm32`, emit a `#[ctor]` function that pushes into a global Vec
- At init time, collect all pushed items into the slice

Problem: `ctor` doesn't work on WASM either (no `.init_array`).

**Option B: Build script codegen**

A build.rs scans the source for `#[distributed_slice(REGISTRY)]` attributes and
generates a static array listing all items. This runs at compile time, not link time.

Problem: Fragile — depends on parsing Rust source.

**Option C: Explicit registration with a macro (recommended)**

On WASM, change the proc macro to emit items into a module-level inventory
collected by a `concat!`-style macro at the crate root:

```rust
// On non-WASM: uses linker sections (current behavior)
// On WASM: emits into a known module, collected by inventory! macro

#[cfg(target_arch = "wasm32")]
macro_rules! distributed_slice {
    ($name:ident) => {
        // Define a mutable Vec behind a OnceCell
        static $name: std::sync::OnceLock<Vec<...>> = std::sync::OnceLock::new();
    };
}
```

Each `#[distributed_slice(REGISTRY)]` item on WASM becomes a `ctor`-like
registration function. Since WASM doesn't have real ctors, we call an explicit
`init_registry()` at interpreter startup that invokes all registration functions.

The proc macro can emit both the linker-section version (non-WASM) and the
explicit-registration version (WASM) behind `#[cfg]`.

**Option D: Static array via proc macro (simplest)**

The proc macro on WASM doesn't use linker sections at all. Instead:
- Each `#[distributed_slice(REGISTRY)]` item is tagged
- A separate proc macro `collect_distributed_slices!()` at the crate root
  scans all tagged items and generates a single static array
- This is the `inventory` crate approach

## Implementation Steps

1. Clone linkme: `git clone https://github.com/dtolnay/linkme`
2. Add to miniR as a path dependency: `linkme = { path = "linkme" }`
3. Remove linkme from vendor (it's now local)
4. On the fork, add a `wasm32` cfg path in the proc macro:
   - `src/distributed_slice.rs` — add `#[cfg(target_arch = "wasm32")]` alternative
   - Generate a static initializer function instead of linker section attributes
5. In miniR's lib.rs, add `#[cfg(target_arch = "wasm32")] linkme::init!();` that
   calls all registration functions
6. Test: `cargo build --target wasm32-unknown-unknown --no-default-features -F minimal`

## File Layout

```
linkme/                    # forked repo (git submodule or vendored)
  src/
    lib.rs
    distributed_slice.rs   # modify for wasm32 cfg
    attr.rs
  impl/                    # proc macro crate
    src/
      lib.rs               # modify: emit wasm32-compatible code
```

## Risks

- dtolnay's linkme is well-maintained; our fork will drift
- The WASM backend is non-trivial — need to understand linkme's internals
- `inventory` crate (by dtolnay) already solves a similar problem but uses
  `ctor` which also doesn't work on WASM
- Alternative: just don't support WASM, or use a manual registry for WASM builds

## Alternative: Manual Registry for WASM Only

If forking linkme is too complex, keep linkme for native and use a manual
registry for WASM:

```rust
#[cfg(not(target_arch = "wasm32"))]
use linkme::distributed_slice;

#[cfg(target_arch = "wasm32")]
fn builtin_registry() -> &'static [BuiltinDescriptor] {
    &[
        // 800+ entries generated by a build script or macro
    ]
}
```

A build script could scan `src/interpreter/builtins/` for `#[builtin]` attributes
and generate the array. This avoids forking linkme entirely.
